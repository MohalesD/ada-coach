Bug resolved Apr 15, 2026
JWT gateway conflict blocking all chat requests

Symptom: every chat message returned "Ada is taking a moment. Please try again." Edge Function logs showed nothing because the code never executed. Root cause: Supabase's gateway had verify_jwt = true in config.toml, which told it to validate JWT tokens before passing requests to the function. The gateway's token format expectations didn't match how the Supabase JS client sends auth headers via fetchWithAuth. Every valid request was rejected at the gate with a 401 before our code ran.

Fix: set verify_jwt = false for all three Edge Functions. Auth is still fully enforced inside the function via requireUser() and requireAdmin(), which call service.auth.getUser(jwt) and validate properly. The gateway check was redundant and broken. Redeployed all functions.

Interview sound bite "We hit a JWT verification conflict where Supabase's gateway was rejecting valid tokens before our Edge Function code could execute. The format the gateway expected didn't match how the client was sending tokens. We resolved it by moving JWT verification inside the function itself — same security, better placement."
Bug resolved Apr 15, 2026
Admin panel race condition — flash redirect before profile loads

Symptom: clicking /admin caused the page to flash briefly then redirect back to chat. The Control Panel appeared for less than a second. Root cause: a frontend race condition. The ProtectedRoute component checked the user's role before auth-context.tsx finished fetching the profile from the database. It saw an undefined role and treated it as "unauthorized," bouncing the user before the real role data arrived.

Fix: updated ProtectedRoute to wait until loading is false before evaluating role. While loading, it shows a spinner instead of redirecting. Once profile data is confirmed, role check runs and admin users pass through.

Interview sound bite "We had a race condition in our route guard where the role check fired before the async profile fetch completed. The fix was ensuring the guard holds in a loading state until the profile is fully resolved — never make an access decision on undefined data."
Security Apr 12, 2026
API key nearly pasted into Claude Code terminal — caught and rotated

During the supabase secrets set step, the Anthropic API key was pasted directly into the Claude Code terminal prompt. Claude Code flagged this immediately as a security concern, noting the key would appear in terminal history and session logs. The key was rotated at console.anthropic.com before any potential exposure could be exploited.

Correct method: create a .env.local file, store the key there, add .env.local to .gitignore, then run supabase secrets set --env-file .env.local. The file is never committed to Git. The key is never typed or pasted in plain text.

Interview sound bite "We store all API keys as Supabase secrets loaded from a gitignored .env.local file, never pasted in the terminal or hardcoded in source. The key never appears in git history, terminal logs, or the browser — it's only accessible server-side via Deno.env.get()."
Decision Apr 12, 2026
Dual-client auth pattern for Edge Functions

Chose the dual-client approach for the /chat Edge Function: one user-scoped Supabase client (with the user's JWT attached) for ownership-scoped database reads, and one service-role client for writes. The user client enforces RLS automatically — if a user tries to read another user's conversation, the database blocks it without any manual code check required. The service role client handles message inserts which need elevated permissions.

Interview sound bite "We use a dual-client pattern in our Edge Functions: a user-scoped client where RLS enforces data ownership automatically, and a service-role client for trusted writes. Defense in depth — the database enforces security, not just our application code."
Decision Apr 12, 2026
pgvector over Pinecone for RAG (Layer 2)

Chose Supabase's native pgvector extension for vector similarity search instead of adding Pinecone as a separate third-party service. pgvector runs inside the same Postgres database, keeping all data in one place, eliminating a second vendor dependency, and reducing architectural complexity for the MVP. Pinecone would add cost, another API key to manage, and network latency between two separate services.

Interview sound bite "We used pgvector for RAG retrieval instead of a dedicated vector database. Keeping vectors in the same Postgres instance as our app data reduced operational complexity, eliminated a vendor dependency, and made the architecture easier to reason about at MVP scale."
Lesson learned Apr 15, 2026
Migration order matters — seed data must come after the user exists

Migration 004 was designed to set mohalesdeis@gmail.com as role='owner' in user_profiles. But the migration ran before the account was created via signup. The UPDATE found zero matching rows and silently did nothing — no error, no warning. The role was never set, so the admin link never appeared.

The fix was running the UPDATE manually via SQL after signing up. For future migrations that depend on specific user data: either make them re-runnable (idempotent), trigger them via a database function after user creation, or document the manual step clearly.

Interview sound bite "We learned that migrations are stateless — they run once at a point in time. A migration that sets a role for a specific user must be designed idempotently, since if it runs before the user exists, it silently no-ops. We now document data-dependent migrations with explicit post-signup steps."
Lesson learned Apr 11–12, 2026
Scope discipline — RAG features folded into Ada's roadmap, not a new product

During Week 2 planning, a detailed People Ops Copilot project plan was generated as an alternative to Ada Coach. The features (RAG, document ingestion, admin upload, evaluation rubrics) were compelling. The temptation to pivot or build both simultaneously was real. Decision: stay with Ada Coach and fold the RAG architecture into Ada's Layer 2 roadmap. The product already had a landing page, brand, and coaching system prompt. Splitting focus would have produced two mediocre products instead of one strong one.

Interview sound bite "When a compelling adjacent idea emerged mid-sprint, we evaluated it against our existing roadmap and found the core features could be incorporated as a future layer rather than a separate product. Scope discipline means recognizing when a new idea strengthens your existing product versus pulls you away from it."
